# Phase 2.1: Feature Iteration — Design Spec

## Overview

Iterate on the XGBoost feature set to improve walk-forward Sharpe ratio above the SMA crossover baseline. The current 6-feature model (Phase 2) achieved Sharpe 0.26 full-period / 0.49 post-2020 — well below the targets. SHAP analysis showed 3 of 6 features (`sma_ratio`, `volume_ratio`, `rsi_14`) contribute near-zero predictive value.

This iteration adds external macro/cross-asset features and uses automated forward feature selection to find the best combination.

**Targets (must beat both):**
- Full-period Sharpe ≥ 0.88
- Post-2020 Sharpe ≥ 1.14

**Phase 2 baseline to improve upon:**

| Metric | Phase 2 XGBoost | SMA Target |
|---|---|---|
| Full-period Sharpe | 0.26 | 0.88 |
| Post-2020 Sharpe | 0.49 | 1.14 |
| Accuracy | 0.44 | — |

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Data sources | Free APIs (yfinance + FRED) | No cost, sufficient for daily features |
| Iteration strategy | Hybrid: candidate pool + forward selection | Avoids guesswork of manual rounds and brute-force cost of exhaustive search |
| Selection metric | Walk-forward Sharpe (full-period) | Same metric as the gate criterion; post-2020 Sharpe used as secondary filter |
| Minimum improvement | ≥ 0.02 Sharpe per added feature | Prevents adding noise features that marginally improve score |
| Base features | log_return_5d, log_return_20d, volatility_20d | SHAP-validated as the only useful features from Phase 2 |

**FRED data retrieval:** Use FRED's public CSV endpoint (`https://fred.stlouisfed.org/graph/fredgraph.csv?id=SERIES&cosd=START&coed=END`) — no API key required.

**Data alignment:**
- All external data merged on SPY's trading calendar via inner join
- FRED yields forward-filled (max 5 business days) to cover holiday gaps and reporting lags before joining; rows with gaps longer than 5 days are dropped
- VIX3M available from ~2007; `vix_term_structure` feature will be NaN before that

**Idempotency:** The script skips re-download if `features_daily.csv` exists and was modified within the last 24 hours. Pass `--force` to re-download.

**Output:** `backtest/data/features_daily.csv` with columns: `date, vix_close, vix3m_close, hyg_close, lqd_close, gld_close, uup_close, tlt_close, eem_close, dgs10, dgs2`

### Candidate Feature Pool

**Existing keepers (3):** Computed from SPY data as in Phase 2.

| Feature | Computation |
|---|---|
| `log_return_5d` | `log(close / close.shift(5))` |
| `log_return_20d` | `log(close / close.shift(20))` |
| `volatility_20d` | `close.pct_change().rolling(20).std()` |

**New candidates (11):** Computed from external data.

| Feature | Computation | Rationale |
|---|---|---|
| `vix_level` | VIX close (raw) | Fear gauge, regime indicator |
| `vix_term_structure` | VIX / VIX3M | Contango/backwardation signals stress vs calm |
| `credit_spread` | `log(HYG/LQD)` 20d change | Price-ratio proxy for credit risk shift (not a true OAS spread — captures mix of credit and duration risk) |
| `yield_curve_slope` | DGS10 - DGS2 | Recession/expansion predictor |
| `yield_10y_level` | DGS10 (raw) | Interest rate environment |
| `gold_momentum` | `log(GLD / GLD.shift(20))` | Safe haven demand trend |
| `dollar_momentum` | `log(UUP / UUP.shift(20))` | Dollar strength / risk appetite |
| `bond_momentum` | `log(TLT / TLT.shift(20))` | Flight to safety signal |
| `em_momentum` | `log(EEM / EEM.shift(20))` | Global risk appetite proxy |
| `vol_of_vol` | `volatility_20d.rolling(20).std()` | Regime instability (note: 40-day effective warmup) |
| `drawdown_depth` | `close / close.rolling(252).max() - 1` | Distance from 1-year high |

## Feature Engineering Changes

`FeaturePipeline` in `feature_engineer.py` is extended:

**New method:** `compute_features_extended(spy_df, external_df) -> DataFrame`
- Computes the 3 base SPY features (unchanged)
- Computes the 11 candidate features from `external_df`
- Returns DataFrame with all 14 feature columns
- Drops NaN rows (warmup period will be ~40 days due to chained rolling windows: 20d volatility + 20d vol_of_vol, plus 20d for momentum features on external data)

**Existing methods unchanged:** `compute_features()` and `compute_features_with_labels()` remain as-is for backward compatibility with the current shadow pipeline. They will be updated to use the winning feature set after selection completes.

**New method:** `compute_features_with_labels_extended(spy_df, external_df) -> DataFrame`
- Same as `compute_features_extended` but adds the forward return label
- Used by the feature selection script

## Forward Feature Selection

`backtest/feature_selection.py` implements greedy forward selection:

### Algorithm

```
base_features = ["log_return_5d", "log_return_20d", "volatility_20d"]
candidates = [list of 11 new features]
selected = base_features.copy()
current_sharpe = evaluate_base()  # initial walk-forward Sharpe
current_post2020 = evaluate_base_post2020()

while candidates:
    best_feature = None
    best_sharpe = current_sharpe
    best_post2020 = current_post2020

    for candidate in candidates:
        test_set = selected + [candidate]
        sharpe_full, sharpe_post2020 = run_walk_forward(test_set)

        if sharpe_full > best_sharpe + 0.02 and sharpe_post2020 >= current_post2020:
            best_feature = candidate
            best_sharpe = sharpe_full
            best_post2020 = sharpe_post2020

    if best_feature is None:
        break  # No candidate improves Sharpe enough

    selected.append(best_feature)
    candidates.remove(best_feature)
    current_sharpe = best_sharpe
    current_post2020 = best_post2020
    log_step(best_feature, best_sharpe, best_post2020)

    if current_sharpe >= 0.88 and current_post2020 >= 1.14:
        break  # Both targets met
```

### Walk-Forward Configuration

Same as Phase 2:
- 2-year training window, 6-month test window, 6-month rolling increment
- Early stopping with last 20% of training window as validation
- Transaction cost of 0.05% per trade

**Important: Sharpe calculation fix.** The Phase 2 `walk_forward_evaluate` computes returns via `pct_change()` on concatenated test-window dates, which produces incorrect returns at window boundaries. The feature selection script must compute returns within each test window separately, then concatenate the strategy returns. This ensures no spurious returns from date gaps between windows.

**VIX3M handling:** `vix_term_structure` is only available from ~2007. When evaluating this candidate, the walk-forward loop only uses windows where VIX3M data exists (train_start ≥ 2005 to ensure 2-year training window has coverage). All other candidates are evaluated across the full date range. This means `vix_term_structure` is scored on fewer windows and its Sharpe may not be directly comparable — the selection algorithm should note this in the output.

## After Selection

Once the best feature set is identified:

1. Update `FeaturePipeline.compute_features()` to use the winning features by default
2. Move `FEATURE_COLUMNS` to `feature_engineer.py` as the single source of truth (currently in `train_model.py` and inline in `main.py` — consolidate)
3. Update `main.py` shadow mode to call `compute_features_extended` with external data
4. Update `download_features.py` to be called as a prerequisite (or integrate into pipeline)
5. Retrain final model via `train_model.py` and save to `models/xgboost_spy.json`
6. If targets met → deploy to shadow monitoring
7. If targets not met → discuss next steps (more features, different model, different horizon)

These updates are NOT part of this spec — they are a manual step after reviewing the selection results.

## Estimated Runtime

Forward feature selection runs up to 11 iterations × ~46 walk-forward windows × model training per candidate per iteration. Estimated 15-30 minutes depending on hardware. Progress is printed per step so you can monitor.
